# 專案背景知識

## 什麼時候需要新增 project background？

當你在使用 `/qa-testcase-generator` 時，發現自己需要先跟 Claude 解釋「這個功能現在長什麼樣子」才能繼續——那段解釋就值得存成 background 檔，下次同功能的 PRD 就不用再說一遍。

**適合存的內容（穩定的既有知識）：**
- 某功能已上線的完整狀態機與業務規則（如：改期的條件、次數限制）
- 特定商品類型的既有限制（如：opendate 商品的特殊行為）
- 某頁面現有的 UI 結構（如：訂單明細頁目前有哪些 CTA）

**不適合存的內容：**
- 這次 PRD 要新增的邏輯（寫在 PRD 就好）
- 還在規劃、尚未穩定的規格
- 一次性專案的背景

## 命名規則

`{feature-name}.md` — 小寫英文，空格用連字號。

Claude 會自動比對 PRD 中的關鍵字與此目錄的檔名，有符合則自動載入。

範例：
- `date-change.md` — 改期功能
- `order-detail.md` — 訂單明細頁

## 使用規則

**新 PRD 規格永遠優先；background 只用於填補 PRD 未提及的既有業務邏輯。**

## 檔案結構範本

```markdown
# {功能名稱} — 背景知識

## 概述
此功能的用途與使用情境。

## 既有業務規則
已上線、穩定的邏輯（如 Gate Condition、狀態機、計算規則）。

## 既有 UI 結構
現有頁面的主要元件與 CTA（有助於判斷新 PRD 的影響範圍）。

## 已知限制
技術或業務上的既有限制。
```
